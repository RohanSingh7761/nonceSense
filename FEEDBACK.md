Feedback for Uniswap:
Building with Uniswap was a strong experience overall but biggist pain point I hit was that quoting can succeed while execution fails, and the failure modes are extremely opaque. A route can quote successfully through the quoter, return what looks like a valid output, and then fail during SwapRouter execution with no meaningful revert data. From a builder perspective, this is one of the biggest pain points because the failure happens in the exact place where developers need clarity most. If the swap fails, I need to know whether it was due to no usable liquidity, invalid tick range, stale pool state, bad token metadata, slippage constraints, or token transfer issues. Instead, the current experience often collapses into a generic “missing revert data” failure, which makes debugging far more difficult than it should be.
A major friction point was testnet reliability, especially on Sepolia. The contracts technically work, but testnet liquidity is so inconsistent that it becomes hard to distinguish protocol bugs from market-quality issues. Pools can exist, quotes can succeed, and the route can still be non-executable because the underlying liquidity is junk, stale, or economically meaningless.
Also The docs are strong if you already understand how Uniswap works, but there are still gaps for builders who are integrating beyond basic swaps.
